# Security

VibeCLI is local-first. Configuration, run state, event logs, plans, and final reports are written under `.vibecli/` in the working repository.

## Secrets

Do not store secrets in `.vibecli/config.yaml`. Provider credentials are referenced by environment variable name only, such as `OPENROUTER_API_KEY`, `LOCAL_API_KEY`, `OPENCODE_SERVER_USERNAME`, and `OPENCODE_SERVER_PASSWORD`.

## Provider Keys

OpenRouter and OpenAI-compatible providers read API keys from `process.env`. The CLI must not print raw provider keys. The OpenCode external adapter may reference server credential environment variables but does not require OpenCode during initialization.

## Command Permission Model

Agent roles define whether shell access is allowed and what scope it belongs to:

- `intake`: no shell, no source writes.
- `repo_scanner`: safe read-only shell.
- `architect`: no shell, no source writes.
- `implementation`: future implementation shell and source writes.
- `test`: test shell and future test writes.
- `security`: scanner shell, no source writes.
- `release_manager`: future git release shell.

Phase 2 does not let agents edit arbitrary project code. Live agents may propose patches, commands, plans, and findings, but VibeCLI saves those proposals as artifacts.

Phase 3 can apply source changes only when all of these are true:

1. The patch came from a VibeCLI run artifact.
2. The patch passed path-policy validation.
3. The run is approved.
4. The user explicitly invokes `vibe apply`.
5. The exact confirmation string is supplied.
6. VibeCLI writes rollback artifacts first.
7. VibeCLI verifies the resulting diff.
8. VibeCLI records the apply in the run ledger.

VibeCLI applies only artifacts generated by VibeCLI runs and approved by the user. It does not allow arbitrary model text to write directly to the repository.

Console mode does not bypass these restrictions. `/apply` and `/rollback` require the same exact confirmation strings as the non-interactive CLI commands.

## Patch Restrictions

Patch proposal paths must be relative, must not contain `..`, and must not target:

- `.env`
- `.env.*`
- `.git/**`
- `node_modules/**`
- `.vibecli/config.yaml`
- `.vibecli/policies/**`
- private keys
- lockfiles

Direct patch application by agents remains disabled.

In Phase 3, patch application is enabled only through:

```bash
vibe apply <run-id> --confirm "APPLY <run-id>"
```

Rollback requires:

```bash
vibe rollback <run-id> --confirm "ROLLBACK <run-id>"
```

Rollback artifacts are written before source files are touched.

## Command Policy

Phase 2 classifies LLM-recommended commands. It does not execute them by default.

Allowed commands are read-only environment and git inspection commands. Install, test, build, and lint commands require approval. Dangerous commands are denied, including recursive deletion, sudo, recursive permission or ownership changes, shell-piped installers, force pushes, package publishing, destructive database commands, shell redirection into `.env`, environment printing, and commands referencing common secret names.

Phase 3 executes only commands classified as `allowed`, only through `vibe commands <run-id> --execute-approved --confirm "EXECUTE COMMANDS <run-id>"`. Commands classified as `requires_approval` or `denied` are skipped.

Inside the console, command execution uses the same policy. Plain text prompts default to dry-run. Live mode prints a warning that provider credits may be spent.

Phase 5 verification is separate from LLM-recommended command execution. `vibe verify <run-id> --confirm "VERIFY <run-id>"` runs only recognized package scripts from `package.json` and refuses dangerous script contents. It does not install dependencies, deploy code, run database migrations, or execute arbitrary agent-suggested commands.

## Secret Content Detection

Patch content is rejected when it appears to include raw API keys, private keys, JWT secrets, credentialed database URLs, or long high-entropy secret-looking values. Safe references such as `OPENROUTER_API_KEY`, `process.env.OPENROUTER_API_KEY`, `${OPENROUTER_API_KEY}`, and `api_key_env: OPENROUTER_API_KEY` are allowed.

## Scanner Gate Foundation

After apply, built-in scanners check protected file touches, secret patterns, env example coverage, package scripts, and test presence. High or critical findings block apply verification. External tools remain planned: `gitleaks`, `osv-scanner`, `semgrep`, `trivy`, and package-manager audits.

Phase 5 adds explicit scanner commands. `vibe scan <run-id>` runs built-in scanners. `vibe scan <run-id> --external --confirm "SCAN <run-id>"` attempts optional external scanners such as `gitleaks`, `osv-scanner`, `semgrep`, `trivy`, and package-manager audit. Missing scanner binaries are recorded as skipped. External scanners are not run automatically after apply, and scanner output is redacted before persistence.

## Streaming Safety

Live streaming records provider lifecycle events and response deltas into the run ledger. It never logs provider API keys, request headers, or environment secret values. Streamed content is still validated against the agent JSON contract after completion before it can become an agent output artifact.

## Repair Cycle Safety

Repair agents propose patches only. They do not apply changes, execute commands, or bypass approval.

`vibe repair <run-id> --live --confirm "REPAIR <run-id>"` is allowed only after a failed verification, scanner, security, or apply gate. Repair cycles are bounded by `budget.max_repair_cycles_per_gate`, and any repair patch must still pass review, patch validation, approval, exact apply confirmation, rollback preparation, and post-apply verification.

## Cost And Budget Safeguards

Model usage is recorded from provider usage fields when available. Cost estimates are marked unknown when pricing is unavailable and must not be treated as exact billing. Max-cost budgets can block additional live agent calls. Cost artifacts must not contain API keys or raw provider credentials.

Phase 6 budget reports are written to `budget-report.json`. `--max-cost-usd` can reduce the run-level budget for a single run. Unknown pricing is surfaced as a warning unless a selected policy requires known pricing.

## Policy-Governed Orchestration

Phase 6 routes agents through local policy and routing configuration. Provider fallback never bypasses patch, approval, apply, verify, scan, or rollback safety. A fallback provider can only produce the same kind of VibeCLI artifacts as the primary provider.

Policy profiles live under `.vibecli/policies/profiles/`. `strict-enterprise` cannot disable exact confirmation. Config validation rejects agents that set both `model` and `model_alias`, aliases that point to missing providers, fallback providers that do not exist, and provider `api_key_env` values that look like raw secrets.

## Ledger Integrity Limits

The run ledger is tamper-evident, not a remote attestation system. It detects local artifact changes but does not prove identity or provide cloud notarization.

`ledger-manifest.json` uses local SHA-256 hashes for run artifacts. Refreshing the ledger requires exact confirmation:

```bash
vibe ledger <run-id> --refresh --confirm "REFRESH LEDGER <run-id>"
```

Ledger verification does not inspect or sign source files outside the run artifact directory.

## Handoff And GitHub Safety

Handoff bundles are redacted review artifacts, not complete forensic archives. They must not include raw API keys, `.env` contents, private key material, raw invalid model outputs, or full unbounded command logs.

VibeCLI can prepare and optionally create a GitHub PR, but it does not bypass local review, apply, verification, scanner, ledger, or confirmation gates.

GitHub PR creation is optional and uses the `gh` CLI. The dry summary mode does not mutate GitHub. PR creation requires exact confirmation. VibeCLI does not push by default, refuses force push, and refuses main/master branches unless explicitly allowed. `--push` requires the exact `PUSH AND CREATE PR <run-id>` confirmation.

Policy exception approval does not mean a change is secure and does not bypass apply confirmation. Approval notes are local integrity records, not legal signatures or identity proof.

## Repository Lifecycle Safety

Phase 8 can prepare branches, commits, PR updates, feedback reports, and merge-readiness reports, but it does not bypass review, apply, verification, scanner, ledger, or confirmation gates.

Branch creation:

- `vibe branch <run-id>` is preview-only.
- `vibe branch <run-id> --create --confirm "CREATE BRANCH <run-id>"` is required to create a branch.
- Branch names are validated for safe git path syntax.
- Branch creation never pushes.

Commit creation:

- `vibe commit <run-id>` is preview-only.
- `vibe commit <run-id> --create --confirm "COMMIT <run-id>"` is required to create a commit.
- Commit creation stages only files recorded as applied by VibeCLI by default.
- Handoff artifacts are not committed by default.
- `.env`, protected paths, private keys, unrelated files, raw model outputs, and full run artifacts are not staged.
- Protected branches are refused unless explicitly allowed by config and command flags.
- No git push is performed by commit commands.

GitHub PR update and sync:

- PR update, comment, and sync require exact confirmations.
- These flows do not push and do not merge.
- No force push is allowed.

Reviewer feedback:

- Local feedback ingestion is read-only and bounded.
- GitHub feedback ingestion requires exact confirmation because it calls `gh pr view`.
- Feedback text is redacted before persistence.
- Feedback ingestion recommends repair planning only; it does not create repair patches automatically.

Merge readiness:

- `vibe merge-readiness` and `vibe merge-check` do not merge.
- The optional GitHub mode is read-only and requires exact confirmation.
- Strict policies should block on failed ledger, missing/failed verification, high/critical scanner findings, unresolved high-risk exceptions, and failed GitHub checks.

VibeCLI can prepare branches, commits, PR updates, and merge-readiness reports, but it does not push, update PRs, commit, or call GitHub mutation commands without exact human confirmation.

## Release Operations Safety

Phase 9 can prepare release packets, changelogs, version plans, release branches, local tags, CI reports, deployment-readiness reports, release approvals, and release-readiness verdicts. Release packets and release reports are redacted, bounded local artifacts and must not include raw API keys, `.env` contents, private keys, CI credentials, or raw provider responses.

Changelog writes require:

```bash
vibe changelog <run-id> --write --confirm "WRITE CHANGELOG <run-id>"
```

Version bump writes require:

```bash
vibe version <run-id> --apply --confirm "APPLY VERSION <run-id>"
```

Release branch creation requires:

```bash
vibe release-branch <run-id> --create --confirm "CREATE RELEASE BRANCH <run-id>"
```

Local tag creation requires:

```bash
vibe tag <run-id> --create --confirm "CREATE TAG <run-id>"
```

Local tag deletion requires:

```bash
vibe tag <run-id> --delete-local --confirm "DELETE TAG <run-id>"
```

Tag creation is local-only. Tag push is not implemented. Deployment is not implemented. Package publishing is not implemented. GitHub release creation is not implemented.

CI ingestion is read-only. `vibe ci <run-id>` does not call GitHub. `vibe ci <run-id> --github --confirm "INGEST CI <run-id>"` uses read-only `gh` commands and does not rerun workflows, dispatch workflows, deploy, publish, or mutate repository state.

Release readiness does not deploy. It only evaluates local run artifacts, release artifacts, CI status, ledger integrity, verification status, scanner status, policy exceptions, approval notes, and local git status.

VibeCLI can prepare release packets, changelogs, version plans, release branches, local tags, CI reports, and deployment-readiness reports, but it does not deploy, publish, merge, push branches, push tags, or create remote releases in Phase 9.

## Signed Provenance And Draft Releases

Phase 10 provenance keys are local-only. Private keys are stored under `.vibecli/keys/`, are gitignored, and must not be committed. `*.private.pem` is also gitignored. VibeCLI never prints private provenance keys and never includes them in handoff bundles, release packets, evidence bundles, logs, or reports.

Provenance is SLSA-inspired local provenance, not certified SLSA compliance. Local signatures are local integrity evidence, not legal identity proof or remote attestation.

Provenance key creation requires:

```bash
vibe provenance key init --confirm "CREATE PROVENANCE KEY"
```

Provenance signing requires:

```bash
vibe provenance <run-id> --sign --confirm "SIGN PROVENANCE <run-id>"
```

Evidence signing requires:

```bash
vibe evidence <run-id> --sign --confirm "SIGN EVIDENCE <run-id>"
```

GitHub release draft creation requires exact confirmation. Draft updates also require exact confirmation. Phase 10 creates drafts only when explicitly requested; it does not publish releases, upload release assets, push tags, push branches, create remote tags, deploy, merge, or publish packages.

VibeCLI can generate and sign local release provenance and optionally create a GitHub draft release, but it does not publish releases, upload assets, push tags, push branches, deploy, merge, or publish packages in Phase 10.

## Remote Attestation Export Safety

Remote attestation submission is disabled by default. Export, transparency, and registry metadata commands are local artifact generators unless `vibe attestation submit` is explicitly invoked.

Remote target tokens are stored as environment variable references only, such as `VIBECLI_ATTESTATION_TOKEN`. Token values are never written to config, logs, receipts, transparency entries, registry metadata, evidence bundles, or release artifacts.

Remote submission requires exact confirmation:

```bash
vibe attestation submit <run-id> --target <name> --confirm "SUBMIT ATTESTATION <run-id> TO <name>"
```

Metadata-only submission is the default. Evidence archive upload is not implemented in Phase 11. Registry push is not implemented in Phase 11. Private provenance keys are never uploaded or bundled.

Remote target URL validation rejects unsafe schemes such as `file:`, `ftp:`, `ssh:`, `gopher:`, `data:`, and `javascript:`. HTTPS is required by default. Private-network targets are rejected by default; localhost targets are allowed only when explicitly configured for local testing.

Local transparency entries are not public transparency claims. Remote transparency claims require a submission receipt.

VibeCLI can generate local attestation exports, transparency entries, and registry metadata, and can optionally submit a redacted metadata payload to a configured HTTPS attestation target with exact confirmation. It does not upload source code, evidence archives, private keys, registry artifacts, branches, tags, releases, deployments, or packages in Phase 11.

## Organization Workflow Safety

Phase 12 organization workflows are local-first. Organization private keys are stored under `.vibecli/org/keys/`, are gitignored, and must not be committed. VibeCLI never prints organization private keys and never includes them in run artifacts, handoff bundles, release packets, evidence bundles, attestation exports, organization evidence exports, audit reports, or logs.

Organization policy bundle signing and multi-reviewer approvals are local integrity records, not legal identity proof. The organization audit log is local and tamper-evident only; it is not remote audit logging.

Remote receipt refresh is read-only and requires exact confirmation before any remote GET:

```bash
vibe receipt-refresh <run-id> --verify-remote --confirm "VERIFY REMOTE RECEIPT <run-id>"
```

Retention planning does not delete evidence in Phase 12. No purge command exists. Evidence export modes are redacted and exclude private keys, `.env` files, raw provider outputs, remote tokens, and unbounded command logs by default.

VibeCLI can generate signed organization policy bundles, local multi-reviewer approval matrices, retention plans, redacted organization evidence exports, remote receipt verification reports, and local organization audit logs. It does not delete evidence, upload source code, upload private keys, bypass release gates, deploy, publish, merge, push branches, push tags, or create remote releases in Phase 12.

## Audit Interoperability Safety

Phase 13 audit exports are redacted local artifacts. Audit schemas are local and configurable, and built-in schemas are evidence mapping templates only.

Compliance bundles are read-only support artifacts, not certification. VibeCLI must not claim SOC 2, ISO 27001, PCI, HIPAA, GDPR, or any legal or regulatory certification.

Reviewer directory imports are local-file only. Raw emails are hashed by default, and storing raw reviewer emails requires exact confirmation. Phase 13 does not integrate with SSO, LDAP, or network identity providers.

Audit report signing uses local organization keys when available, otherwise local provenance keys. Private keys are never included in audit exports, compliance bundles, auditor handoff bundles, handoff bundles, evidence bundles, attestation exports, logs, or release artifacts.

VibeCLI can generate local audit evidence maps, signed audit report exports, read-only compliance check bundles, reviewer directory imports, and auditor handoff bundles. These are audit-support artifacts only; they do not certify legal/regulatory compliance and they do not upload evidence, source code, private keys, secrets, or logs to remote services in Phase 13.

## Evidence Lifecycle Safety

Phase 14 evidence lifecycle automation is local-first. Inventories, retention previews, archives, retention ledger events, legal hold records, compaction reports, compact bundles, and cross-run lifecycle reports are written under `.vibecli/`.

Evidence archives are local only and require exact confirmation:

```bash
vibe evidence-archive <run-id> --create --confirm "ARCHIVE EVIDENCE <run-id>"
```

Private keys, organization keys, provenance keys, `.env` files, raw provider outputs, and unbounded command logs are excluded from lifecycle archives and compact bundles by default. Original evidence is not deleted, moved, or rewritten by archive or compaction commands.

Retention enforcement is preview-only. Purge/delete is not implemented in Phase 14, and no purge command exists. Legal hold is metadata only and is not legal advice. The retention ledger is local tamper-evident evidence only; it is not immutable storage, legal notarization, or remote attestation.

VibeCLI can inventory evidence, preview retention enforcement, create local redacted archives, record signed retention ledger events, mark legal hold metadata, and generate compaction reports. It does not delete evidence, purge artifacts, upload evidence, upload source code, upload private keys, deploy, publish, merge, push branches, push tags, or claim legal/regulatory compliance in Phase 14.

## Governed Evidence Disposal Safety

Phase 15 disposal is local-only and exact-confirmed. Disposal commands may delete only approved candidate files under `.vibecli/runs/<run-id>` after eligibility checks, a disposal plan, pre-delete checks, and `--confirm "DELETE EVIDENCE <run-id>"`.

Disposal never deletes source files, project config, package files, lockfiles, docs, `.git`, evidence archives under `.vibecli/evidence-archive`, provenance keys under `.vibecli/keys`, organization keys under `.vibecli/org/keys`, `.env` files, global retention ledgers, organization audit logs, remote evidence, cloud storage, branches, tags, releases, deployments, packages, or registry artifacts.

Legal hold blocks disposal. Archive verification is required when configured. Disposal receipts and deleted-artifact records are local evidence only; they are not immutable deletion proof, legal advice, or certified retention compliance. Recovery is best-effort only and is not guaranteed.

VibeCLI can plan, attest, approve, precheck, and execute exact-confirmed local disposal of eligible run-evidence files only. It does not delete source code, archives, private keys, env files, remote evidence, cloud storage, branches, tags, releases, deployments, packages, or registry artifacts in Phase 15.

## Secret-Safe Logging

VibeCLI must not print raw provider API keys. Provider configuration stores environment variable names only. Errors may mention missing environment variable names but must not include their values.

Console history, when written, is bounded to the latest 100 entries and records command metadata only. It must not store raw API keys or unredacted secret-bearing command output.

## Prompt Injection

Repository files are data, not instructions. Only VibeCLI policies and agent system prompts define behavior. Agent prompts explicitly tell live agents to treat repository content as context rather than executable instructions.

## Phase 16 Dogfood and Beta Safety

VibeCLI Phase 16 adds local dogfood QA and beta readiness checks. Dogfood fixture repositories are isolated under `.vibecli/dogfood/fixtures` and dry-run workflows do not call live providers by default.

Live provider smoke requires exact confirmation:

```sh
vibe live-smoke --provider <provider> --model <model> --confirm "RUN LIVE SMOKE"
```

External scanner execution is disabled by default. `scanner-check --run-safe` requires exact confirmation and runs only safe local version/help checks.

Package checks never publish packages, log in to registries, create releases, deploy, push branches, or push tags.

Phase 16 beta readiness is a local quality gate, not production certification.

Statement: VibeCLI Phase 16 adds local dogfood QA and beta readiness checks. It does not publish packages, deploy, push branches, push tags, create releases, upload evidence, upload source code, or call live providers unless exact confirmation is provided.

## Phase 17 Beta Release-Candidate Gates

Phase 17 beta RC checks are local quality gates, not production certification.

- Live RC smoke requires exact confirmation: `RUN LIVE RC SMOKE`.
- Dogfood apply smoke mutates only isolated fixtures under `.vibecli/dogfood`.
- Package install checks never publish and never log in to a registry.
- Scanner install guidance does not install scanners.
- Warning acceptance is local risk tracking, not risk removal.
- No publish, deploy, push, release, source upload, evidence upload, or private key upload occurs in Phase 17.

Statement: VibeCLI Phase 17 adds local beta release-candidate gates, warning triage, package install validation, dogfood patch-apply smoke, and beta trial packs. It does not publish packages, deploy, push branches, push tags, create releases, upload evidence, upload source code, or call live providers unless exact confirmation is provided.
